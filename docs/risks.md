# Temporal 已知风险与应对

## 范围

本文档记录当前 preview 修复与运行时状态机调整后的已知风险、影响和建议操作，覆盖 UI 一致性、监听控制语义和编码稳定性。

## 风险 1：通道容量上限

- 描述：当前按 `4` 通道固定映射 source，本轮未改变录音通道容量模型。
- 影响：超过 `4` 个 source 时，超出部分不会进入录制会话。
- 现状：已通过 `recordingSourceCount` 和 `recordingSessions` 对齐，相关测试保持通过。
- 操作建议：现场多源场景需提前确认优先 source，并区分“可见 source”与“可录制 source”。

## 风险 2：网络抖动导致超时停录

- 描述：`SST` 更新延迟仍可能触发 inactive timeout，本轮未重构运行时订阅架构。
- 影响：会话被提前关闭，形成多个碎片文件。
- 现状：已覆盖 timeout refresh 与重连恢复测试，但高抖动链路仍需人工观察。
- 操作建议：高抖动网络下适当提高 timeout，并在手工验收时重点关注录制连续性。

## 风险 3：远端日志可见性依赖 SSH 稳定性

- 描述：SSH 波动会影响日志拉取与状态观察；主按钮停止后也会保留 SSH 连接。
- 影响：运维判断启动状态变慢，容易误判远端是否已真正停止。
- 现状：已实现轮询与失败提示，且停止主按钮不会主动断 SSH。
- 操作建议：先确认 SSH 稳定，再执行流监听；必要时单独观察日志尾部确认远端状态。

## 风险 4：磁盘空间耗尽

- 描述：长时间录制可能占满本地磁盘。
- 影响：写盘失败或文件损坏。
- 现状：暂未实现配额控制。
- 操作建议：定期清理 [recordings](recordings)，并做容量监控。

## 风险 5：配置漂移

- 描述：本地配置与远端 `odaslive` 配置不一致时，可能出现界面可启动但无有效流数据。
- 影响：端口可达但无有效数据流，或 preview/runtime 观察结果不一致。
- 现状：已有模板配置，但无自动校验。
- 操作建议：联调前逐项核对主机、端口、命令，并区分 preview 样本数据与 runtime 真实数据。

## 回退策略

1. 若 preview 或 UI 文档再次出现乱码，优先回退到 UTF-8 正常版本后再重新 patch，避免在乱码文件上增量修改。
2. 若主按钮与监听按钮行为异常，优先验证“监听可独立开启”和“主按钮停止先停监听”的状态机约束是否被破坏。
3. 若图表与 3D 不同步，优先检查两者是否仍然共享同一份 `trackingFrames` 时序源。
4. 若问题持续，使用 [docs/runbook.md](docs/runbook.md) 逐步重启流程，并回退到最近稳定配置与网络路径。
