## Assumptions

- Preview 仍然是稳定的演示数据路径，而不是真实采集回放路径。
- Preview 与 runtime 最终应共享一套规范化 tracking schema，
  即便当前 preview 数据仍由 Python fixtures 以确定性方式生成。
- 这一阶段仍优先保证 UI 一致性与状态机一致性，
  暂不优先处理异常场景逼真度。
- 逐帧动态 `energy` 是下一步驱动右侧边栏刷新逻辑的必要模型能力，
  但本轮代码尚未实现。
- 远端 ODAS 继续采用 `ODAS -> Temporal` 的主动推流方向。
- SSH 连接仍由远端 ODAS 启动路径隐式触发，
  而不是新增独立连接按钮。
- 远端 Linux 提供标准 `sh` 和 `setsid`，
  允许通过 Paramiko session channel 持久持有非交互 `sh` control session。
- `odas.command` 允许指向 wrapper script，
  但产品语义要求 stop 仍覆盖 wrapper 与实际 `odaslive` 子进程的完整生命周期。
- listener-button 的启用态应反映活的 SSH control channel，
  而不是仅反映历史连接结果。

## Unresolved Risks

- Preview 文档已经把逐帧动态 `energy` 记为下一步模型能力，
  因此当前实现与已落地的 bridge 行为之间仍存在差距。
- 真实 preview 窗口下的手工端到端验证还需要再做一轮，
  尤其是右侧边栏数字刷新和筛选联动。
- 远端 odaslive 的端到端验证尚未在最近一轮 preview 跟进决策后重新执行。
- 工作区仍有无关未提交变更：
  - `config/odas.example.toml`
  - `specs/in-progress.md`
- `src/temporal/app.py` 仍包含部分历史中文文案编码漂移。
  虽不影响本次逻辑修复，但后续做 UI 文案整理时需要统一清理。
- `docs/runbook.md` 仍保留“先停监听，再停远端 odaslive”的旧操作顺序表述，
  与最新独立生命周期 spec 不一致，后续应同步。
- 虽然 stop 已改为 process-group 语义，
  仍建议后续补一条真实 wrapper 进程树的集成验证，
  确认目标远端环境中的行为稳定。
